## Context

当前 `ActivityBar` 位于共享应用壳层，采用窄栏纯图标按钮 + `UTooltip` 的方式呈现导航项。现有实现已经具备按路由高亮、按 `group` 分组（top / bottom）、按项目存在与否禁用导航等逻辑，但缺少品牌头部，也要求用户通过悬浮才能识别入口含义。

这次改动只涉及渲染层壳组件与其测试，不修改 `activityBarItems` 的路由语义、分组语义，也不改动 store、IPC、共享类型或页面结构。资源文件已存在于 `frontend/public/icon.svg`，在 Electron 打包后的 `file://` 场景下，必须使用 `${import.meta.env.BASE_URL}icon.svg` 构造图片地址，不能写死根路径。

## Goals / Non-Goals

**Goals:**

- 让 Activity Bar 导航标签常驻可见，降低新用户理解成本。
- 在顶部加入 FylloCode 品牌 icon，形成“品牌 / 菜单 / 设置”三段布局。
- 保持现有路由跳转、高亮、项目约束逻辑不变。
- 用测试覆盖新的结构与关键文案，避免后续回归到纯 icon 导航。

**Non-Goals:**

- 不新增新的导航入口或改变 `activityBarItems` 注册表。
- 不修改路由守卫、默认路由、settings 页面能力或其他壳层组件。
- 不引入新的图片资源处理方式或新依赖。

## Decisions

1. 将 Activity Bar 宽度从纯 icon 窄栏扩展为可容纳“图标 + 两字到四字文案”的固定宽度侧栏。
   原因：如果仍保持原始窄宽，标签只能换行堆叠或被截断，反而降低可读性。固定宽度可以让三段结构更稳定。
   备选方案：继续使用 tooltip，只给当前选中项显示文案。放弃原因是仍无法满足“无需 tooltip 即可理解”的目标。

2. 顶部品牌区直接在 `ActivityBar.vue` 内使用 `<img>` 加载 `${import.meta.env.BASE_URL}icon.svg`。
   原因：该资源只服务于壳层导航，不值得为此增加额外配置或抽象；`BASE_URL` 明确兼容 `file://` 打包场景。
   备选方案：将 icon 转为组件或用 CSS background。放弃原因是对当前仓库没有额外收益，且不如显式 `img` 清晰。

3. 去掉导航按钮的 `UTooltip` 包裹，按钮内部改为垂直布局显示图标和标签。
   原因：需求要求标签直接替代 tooltip；将标签与按钮绑定可保持点击区完整，也更易于统一激活态样式。
   备选方案：在按钮外单独渲染标签。放弃原因是会拆散视觉与交互归属，增加对齐复杂度。

4. 测试从“只校验按钮数量与高亮”扩展到“校验品牌图标、标签文案、分区结构与高亮”。
   原因：这次改动的核心是视觉结构而非业务逻辑，测试必须覆盖新的 DOM 语义，否则后续容易被还原。

## Risks / Trade-offs

- [侧栏变宽后影响主区域横向空间] → 保持改动只在 `ActivityBar` 固定宽度内，且仅扩大到容纳短标签的最小必要宽度。
- [文案常驻后按钮高度增加，可能削弱紧凑度] → 使用小字号和紧凑垂直间距，保证顶部菜单仍能完整容纳。
- [品牌 icon 在打包后路径失效] → 强制使用 `${import.meta.env.BASE_URL}icon.svg`，并通过测试断言 `src` 组成。
